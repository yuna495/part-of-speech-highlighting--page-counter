# コメント／ドキュメンテーション統一ルール

- **目的**: 可読性を保ち、機能差分を生まないためにコメント表記を統一する。
- **言語**: 日本語。必要に応じて英語用語を併記してもよい。
- **書き方の基本**
  - 関数・クラス・公開 API には JSDoc を付与する。必要最小限で「何を／なぜ」を記す。
    - `@param {Type} name 説明`
    - `@returns {Type} 説明` もしくは `@returns {void}`
    - 非同期関数は `@async` を付ける
  - 簡単な補足・処理理由: `//` 1 行コメントで記載。複数行の場合も `//` を連ねる。
  - 未実装・改善タスク: `// TODO: ...`
  - ブロックコメント `/* ... */` は使用しない（JSDoc を除く）。
- **内容の粒度**
  - コードから自明なこと（ループ・代入など）にはコメントを付けない。
  - 処理方法より「意図・前提・制約」を優先して書く。
- **句読点・表記**
  - 句読点は「、」「。」を使う。半角カンマ・ピリオドで文章を区切らない。
- **ファイル先頭**
  - 先頭に 1 行でファイルの目的を記す（必要に応じて）。既存の先頭コメントがある場合はその形式を踏襲してよい。
- **その他**
  - 実装コードは変更せず、コメントのみで調整することを基本とする。
  - エンコードは UTF-8 を前提とし、文字化けを避ける。
